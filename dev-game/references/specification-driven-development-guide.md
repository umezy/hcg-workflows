# Unity カジュアルゲーム 仕様駆動開発ガイド

## 概要

本ドキュメントは、Unityでカジュアルゲームを仕様駆動開発(Specification-Driven Development)で作成するための手順を定義する。
仕様を先に確定させ、それに基づいて設計・実装・テストを行うことで、手戻りを最小化し品質を担保する。

---

## 開発フロー全体像

```
1. ゲーム企画書作成
2. 機能仕様書作成
3. 技術仕様書作成
4. テスト仕様書作成
5. タスクリスト作成
6. 実装
7. テスト・検証
8. リリース
```

---

## Phase 1: ゲーム企画書の作成

**成果物:** `docs/01-game-design.md`

以下を定義する:

- ゲームタイトル
- ジャンル・コンセプト（1行で表現できるゲームの核）
- ターゲットプラットフォーム（WebGL / モバイル / PC）
- ターゲットユーザー
- コアループ（プレイヤーが繰り返す基本行動）
- 勝利条件・敗北条件
- ゲームの面白さの源泉（何が楽しいのか）
- 参考ゲーム・インスピレーション元

---

## Phase 2: 機能仕様書の作成

**成果物:** `docs/02-functional-spec.md`

ゲームの「何ができるか」を網羅的に定義する。

### 2.1 画面一覧・画面遷移図

| 画面名 | 説明 | 遷移先 |
|--------|------|--------|
| タイトル画面 | ゲーム開始前の画面 | ゲーム画面 |
| ゲーム画面 | メインのゲームプレイ画面 | リザルト画面 |
| リザルト画面 | スコア表示・リトライ | タイトル画面 |

### 2.2 ゲームメカニクス仕様

各メカニクスについて以下を記述する:

- 入力（プレイヤー操作）
- 処理（ゲーム内部のロジック）
- 出力（画面表示・フィードバック）
- 境界条件・エッジケース

### 2.3 スコア・難易度仕様

- スコア計算式
- 難易度曲線（時間経過に伴う変化）
- パラメータ一覧と初期値

### 2.4 UI/UX仕様

- 各画面のレイアウト（ワイヤーフレームレベル）
- ボタン・テキストの配置と機能
- アニメーション・演出の概要

---

## Phase 3: 技術仕様書の作成

**成果物:** `docs/03-technical-spec.md`

ゲームを「どう作るか」を定義する。

### 3.1 プロジェクト構成

```
Assets/
├── Scripts/
│   ├── Core/           # ゲームマネージャー、状態管理
│   ├── Gameplay/       # ゲームプレイロジック
│   ├── UI/             # UI制御
│   └── Data/           # データクラス、ScriptableObject
├── Prefabs/            # 再利用可能なオブジェクト
├── Scenes/             # シーンファイル
├── Materials/          # マテリアル
├── Sprites/            # 2D画像
└── Audio/              # 効果音・BGM
```

### 3.2 クラス設計

主要クラスとその責務を定義する:

| クラス名 | 責務 | 依存先 |
|---------|------|--------|
| GameManager | ゲーム全体の状態管理 | - |
| ScoreManager | スコアの計算・保持 | GameManager |
| UIController | UI表示の更新 | GameManager, ScoreManager |

### 3.3 状態遷移図

ゲームの状態（State）とその遷移条件を定義する:

```
[Init] → [Title] → [Playing] → [GameOver] → [Result] → [Title]
```

### 3.4 データ設計

- 保存が必要なデータ（ハイスコア等）
- 保存方式（PlayerPrefs / JSON）
- ScriptableObjectで管理するパラメータ

## Phase 4: テスト仕様書の作成

**成果物:** `docs/04-test-spec.md`

実装前にテスト項目を定義する（仕様駆動の要）。

### 4.1 単体テスト

機能仕様の各項目に対応するテストケースを作成する:

| テストID | テスト対象 | 入力 | 期待結果 |
|---------|-----------|------|---------|
| UT-001 | スコア加算 | ターゲットをタップ | スコアが+10される |
| UT-002 | タイムアウト | 制限時間0 | GameOver状態に遷移 |

### 4.2 統合テスト

画面遷移・複数コンポーネント連携のテスト:

| テストID | テスト対象 | 手順 | 期待結果 |
|---------|-----------|------|---------|
| IT-001 | ゲーム開始→終了フロー | スタート→プレイ→タイムアウト | リザルト画面にスコア表示 |

### 4.3 受け入れテスト

機能仕様を満たしているかの最終確認チェックリスト。

## Phase 5: タスクリスト作成

**成果物:** `docs/05-task-list.md`

仕様書・テスト仕様書の内容からタスクを洗い出し、実装前に作業計画を立てる。

### 5.1 タスクの洗い出し

- 機能仕様書・技術仕様書の各項目をタスク単位に分解する
- テスト仕様書のテストIDと対応づける

### 5.2 優先順位・依存関係の整理

- タスク間の依存関係を明確にする
- 実装順序を決定する（依存されるタスクを先に実施）

### 5.3 タスクリストのフォーマット例

| タスクID | タスク内容 | 対応テストID | 依存タスク | ステータス |
|---------|-----------|-------------|-----------|-----------|
| TASK-001 | GameManager の状態管理実装 | UT-002 | - | 未着手 |
| TASK-002 | スコア加算ロジック実装 | UT-001 | TASK-001 | 未着手 |

## Phase 6: 実装

仕様書・タスクリストに基づいて、Unity MCP ツールで実装を進める。

### Unity MCP ツールの活用

- スクリプト: `create_script` → `validate_script`
- シーン構築: `manage_gameobject` + `manage_components`
- UI: `manage_ui`
- Prefab: `manage_prefabs`
- マテリアル: `manage_material`
- 演出: `manage_animation` / `manage_vfx`
- 確認: `refresh_unity` + `read_console`
- テスト: `run_tests` + `read_console`

### 1タスクの実装サイクル

各タスクは以下の順で進める:

1. **実装** — Unity MCP で機能コードを作成
2. **テストコード作成** — テスト仕様書（Phase 4）の対応テストIDに基づきテストコードを作成
3. **テスト実行** — `run_tests` で対象テストを実行し合格を確認
4. **L1/L2 検証** — コンパイルエラー・ランタイムエラーがないことを確認
5. **コミット** — 実装コード＋テストコードをセットでコミット
6. **タスクリスト更新** — ステータスを `✅完了` に更新

### 実装の原則

1. **仕様書にない機能は作らない** — スコープを守る
2. **1タスク1コミット** — 実装コードとテストコードをセットでコミットし、テストIDと結果を含める
3. **パラメータはハードコードしない** — ScriptableObject または定数クラスで管理
4. **テストIDがコミットメッセージにある = テストが合格済み** — テスト未実行のまま PASS を記載しない

### 実装順序（推奨）

1. プロジェクト構成・フォルダ作成
2. GameManager（状態管理の骨格）+ DebugManager
3. コアゲームメカニクス
4. スコアシステム
5. UI実装
6. 演出・エフェクト
7. サウンド
8. データ保存（ハイスコア等）

---

## Phase 7: 統合検証・受け入れテスト

Phase 6 で各タスクごとにテストコードは作成済み。Phase 7 では全テストの一括実行と受け入れテストを行う。

### 7.1 全テスト一括実行

- `run_tests` で EditMode（UT + RT）・PlayMode（IT）の全テストを一括実行
- `read_console` で結果を確認、不合格項目は修正後に再テスト

### 7.2 受け入れテスト・プレイテスト

- McpRemoteControl による受け入れテスト自動化
- `manage_editor` で Play Mode 実行、手触りを確認
- 難易度バランスの調整（パラメータ変更のみで対応）

---

## Phase 8: リリース

- `manage_editor` でビルド設定の確認（解像度、プラットフォーム固有設定）
- ビルド実行・動作確認
- 配布（WebGL → itch.io / GitHub Pages など）
